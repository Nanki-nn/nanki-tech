## 先确定任务边界

可维护的 Agent 工作流，第一步不是多接几个工具，而是把任务边界说清楚。一个好的边界应该回答三个问题：它负责什么、它不负责什么、失败时应该把控制权交还给谁。

如果边界不清，Agent 很容易把“继续尝试”误当成“解决问题”。工程上更稳妥的做法，是为每个任务定义可观察的完成条件，例如文件已生成、测试已通过、数据已写入，或者用户已经确认下一步。

## 把工具调用当作接口设计

工具不是魔法按钮，而是外部系统接口。每次调用都应该有明确输入、预期输出和失败处理。对于会修改文件、发送请求或产生外部副作用的工具，更需要把权限和确认点设计清楚。

一个实践经验是：让 Agent 优先读取上下文，再做最小改动，最后验证结果。这个顺序能降低误改和过度实现的概率。

## 管理上下文，而不是堆上下文

上下文越多不一定越好。长期运行的工作流需要区分“当前必须知道的信息”和“以后可能有用的信息”。前者进入当前推理，后者应该沉淀到文档、状态文件或可查询索引里。

当上下文被压缩时，最重要的是保留目标、约束、已完成动作、未解决风险和下一步。少量高质量状态，比完整聊天记录更可靠。

## 失败恢复要显式

Agent 系统一定会遇到工具失败、网络超时、文件冲突和需求变化。可维护性来自显式恢复策略：重试几次、什么时候换路径、什么时候停止并说明原因。

我更倾向于把失败恢复写成可测试的分支，而不是藏在提示词里的“尽量处理”。这样系统行为更容易复盘，也更容易让人信任。
